# Submission Video: From Complaint to Proof

Target length: 72 seconds. Record the product, not a talking head. Use hard cuts, readable browser zoom, and one continuous civic example from evidence to public proof.

## Voiceover

> A broken drain can be reported in seconds. Then the complaint can disappear into a queue no citizen can inspect.
>
> Nagarik Signal changes the unit from complaint to proof.
>
> A report keeps the public evidence, strips photo metadata, publishes only an approximate location, and anchors evidence, metadata, and location commitments to a Solana Issue PDA.
>
> Other citizens can say, "I saw this too." Each accepted signal creates a Verification PDA; repeat verification from the same signer is rejected.
>
> But do not trust the interface. On a live devnet record, Nagarik Signal recomputes the hashes and compares them with the Solana account in public.
>
> Steward updates create StatusUpdate PDAs, so submitted, verified, in progress, and resolved remain one inspectable timeline.
>
> The dashboard makes neglect measurable by ward: what is still open, how long it has been ignored, and what was resolved with proof.
>
> Demo records are clearly labeled. Live devnet proof stays separate. No token. No rewards. Just public memory for public problems.
>
> Janamat shows what citizens think. Nagarik Signal shows what citizens proved.

## Capture Checklist

- Record at `1920x1080`, 30 fps.
- Pre-open `/`, `/report`, one clean `indexed_devnet` issue, `/dashboard`, and the Solana Explorer program page.
- Keep the demo label visible when a seeded record appears.
- Use only an `indexed_devnet` issue for the Solana verification shot.
- Use a fresh browser session for the citizen-signal shot and capture a genuine accepted verification.
- Warm the proof check once before recording so RPC latency does not interrupt the take.
- Record write operations from the stateful build, not a read-only Vercel review deployment.
- Hide bookmarks, notifications, local paths, credentials, and unrelated tabs.
- Keep cursor movement deliberate and avoid decorative transitions.
- Leave the final repository and live-demo links on screen for at least three seconds.
